## Fixture history

### G2 redesign (3 iterations)

G2 was originally aimed at GEO1 (`['GENERALIZE', 'kANON']` pipeline for
multi-level geographic QIs). The fixture's gate conditions were met correctly:
city (60 unique = fine), region (10 = coarse), income (continuous to bring
cat_ratio below 0.70).

**Iteration 1:** GEO1 gate matched but `select_method_suite` rejected the
pipeline — GENERALIZE was missing from `METRIC_ALLOWED_METHODS`. The fixture
fell through to QR4_Widespread. Diagnosis at the time: "GEO1 is dead code."
Wrong — config omission, not dead gate.

**Iteration 2:** G2 retargeted to QR4_Widespread (accepted the false diagnosis).
Passed verification.

**Iteration 3 (Fix 0):** GENERALIZE added to all 4 metric lists. G2 retargeted
back to GEO1. Now passes verification with `pipeline=['GENERALIZE', 'kANON']`.

This sequence is Spec 16 input: it shows where the rule chain's behavior is
hard to predict from the spec alone. The config-layer rejection was silent —
no log, no error, just fallthrough.

### G7 redesign (2 iterations)

G7 was originally aimed at CAT2_Mixed_Categorical_Majority (`['NOISE', 'PRAM']`
pipeline under l_diversity). CAT2 is gated to l_diversity metric, but NOISE is
blocked for l_diversity. The pipeline always matches then gets rejected.

**Iteration 1:** CAT2 found unreachable (same NOISE/l_diversity contradiction
as DYN_CAT). Retargeted to LOW2_Continuous_Noise.

**Iteration 2:** Passed verification. No further changes after Fix 0 (GENERALIZE
fix doesn't affect CAT2's NOISE issue).

## Change history

### Fix 0 (2026-04-20): GENERALIZE added to METRIC_ALLOWED_METHODS

`GENERALIZE` and `GENERALIZE_FIRST` were missing from `METRIC_ALLOWED_METHODS`
in `config.py` since inception. Both are non-perturbative recoding methods
(merge categories, preserve frequency counts) and are compatible with all
four risk metrics.

**Impact:** Three rules were silently blocked for all metrics:
- **GEO1** (`['GENERALIZE', 'kANON']` pipeline) — geographic multi-level
- **RC4** (`['GENERALIZE', 'kANON']` pipeline) — risk concentration bottleneck
- **QR0** (`GENERALIZE_FIRST` single method) — infeasible QI space preprocessing

**Fixture changes after Fix 0:**
- G2: Previously fell through to `QR4_Widespread`. Now correctly hits
  `GEO1_Multi_Level_Geographic`.
- All other fixtures: no change in rule mapping.

**Systemic prevention:** `tests/test_method_metric_coverage.py` added — asserts
every method name in any rule's `method` field or `pipeline` list appears in
`METRIC_ALLOWED_METHODS` for at least one metric.

**Implication for past harness results:** The 80-run harness outcomes are not
wrong, but some runs reached their eventual rule via fallthrough from a rule
that should have matched. The audit needs to account for this.

### Category E: Preempted-always under current metric algebra

RC2, RC3, and RC4 are structurally preempted by RC1. The backward
elimination contribution metric has a 50% floor: `contrib ≈ (1 - 1/k) * 100`
for a QI with cardinality k, so the minimum contribution for any QI with
k >= 2 is 50%. This means `classify_risk_concentration()` always assigns
the `dominated` pattern (top_pct >= 40%), and RC1 fires first.

RC2, RC3, and RC4 all require `top_pct < 40%`, which the metric cannot
produce. This is not a harness coverage gap — it is an algebraic property
of the leave-one-out contribution metric.

See `docs/investigations/spec_16_readiness_rc_family_preemption.md` for full
algebra, empirical verification on adult dataset, and design implications.

| Rule | Gate requires | Why unreachable | Fixture |
|---|---|---|---|
| RC2_Risk_Concentrated | `concentrated` (top2 >= 60%, top < 40%) | top always >= 50% | None needed |
| RC3_Risk_Spread_High | `spread_high` (3+ HIGH, top < 40%) | top always >= 50% | None needed |
| RC4_Single_Bottleneck | `balanced` + n_high==1 + n_other>=3 | `balanced` never assigned; also needs 15-39% window | G9 (injected var_priority) tests rule-selection logic |

## Dormant-rule taxonomy for Spec 16

The audit should distinguish four kinds of "not firing":

1. **Config-blocked:** Gate matches, pipeline/method valid, but config-layer
   filter (`METRIC_ALLOWED_METHODS`) rejects. Silent rejection, no log.
   *Example: GEO1 before Fix 0.*

2. **Gate-matches-but-preempted:** Data shape satisfies the gate, but a
   higher-priority rule fires first. The rule is reachable in theory but
   shadowed in practice.
   *Example: DP1_Outliers preempted by DYN_Pipeline on the same data.*

3. **Preempted-always:** Gate conditions cannot coexist with a higher-priority
   rule matching, given the algebraic properties of the metrics consumed.
   Structurally impossible, not probabilistic.
   *Example: RC2/RC3/RC4 — require top_pct < 40%, but 50% floor means
   RC1's dominated pattern always matches first.*

4. **Gate-doesn't-match-on-any-harness-dataset:** The condition is valid code
   but no dataset in the harness produces the required feature shape. This is
   a weaker claim than "no realistic data triggers it" — it may just mean the
   harness is missing that data shape.
   *Example: DATE1 — no temporal fixtures in the harness.*
